# Change Log - 2026-06-04 07:52:15

## 本轮问题背景

本轮主要围绕 HUGSIM actor-learner 训练链路做稳定性修复，目标是解决以下几类问题：

1. 多 actor 并发运行时，HUGSIM FIFO 会话目录和命名管道互相冲突。
2. actor 数量随训练推进逐渐减少，部分 actor/runner 卡在 FIFO 读写两端。
3. 某些 episode 在上层被判定 done 后，底层 FIFO session 没有同步关闭，后续 reset/plan 状态错位。
4. learner 训练阶段 DataLoader worker 访问 GPU tensor，引发 `CUDA error: initialization error`。
5. HUGSIM 子进程状态不透明，难以定位卡在 reset、step 还是 waiting_plan。

当前观察结论是：训练已恢复正常推进，reset 也恢复正常。

---

## 主要框架修复

### 1. `framework/env_wrapper/fifo_io.py`

**修复内容：**

- 将 FIFO 传输协议从“依赖 EOF 判定消息结束”改为**8 字节长度前缀 + payload** 的 framed protocol。
- `write_fifo_payload()` 先写入长度，再写入完整 pickled payload。
- `read_fifo_payload()` 按固定长度读取完整消息，不再依赖对端关闭 FIFO。
- 保留超时和 peer-process 存活检测，异常统一抛 `FifoCommunicationError`。

**解决的问题：**

- 修复 actor 与 HUGSIM runner 双方都保持 FIFO 打开时，读取端因等不到 EOF 而永久阻塞的问题。
- 降低多进程 FIFO 边界不清导致的协议错位和“假死”概率。

---

### 2. `framework/env_wrapper/hugsim_adapter.py`

**修复内容：**

- 为每个 HUGSIM session 输出目录增加唯一 `session_tag`，目录结构变为：
  - `<output_root>/<scenario>/<actorX_workerY_rankZ_cudaN>/`
- `HUGSIMFifoClient.start()` 启动前清理旧的 `obs_pipe`、`plan_pipe`、`status.json`，避免复用脏会话文件。
- 新增/整理 `fifo_step_timeout_s`，将 step 级别 FIFO 超时与整体启动超时分开控制。
- 当 FIFO step 读写超时时，将该 step 转成 `truncated=True` 的终止样本，并记录 `fifo_timeout_error`。
- 关键修复：**当上层 wrapper 在碰撞、short route、timeout 等条件下判定 episode 已结束时，立即 `close()` 底层 FIFO client**。

**解决的问题：**

- 修复不同 actor 共享同一 scenario 输出目录时，FIFO 文件互删/互抢的问题。
- 修复“上层判 done，但底层 HUGSIM runner 仍停在 waiting_plan，导致下一次 reset 错位”的 session 生命周期 bug。
- 修复部分 reset 慢/假卡死的根因之一：旧 session 未关闭，新 session 又开始 reset。

---

### 3. `framework/env_wrapper/hugsim_fifo_runner.py`

**修复内容：**

- 扩展 `_write_status()` 输出内容，持续写入 `status.json`。
- 状态拆分为：
  - `state`: `starting` / `running` / `stopped` / `completed` / `error`
  - `phase`: `boot` / `loading_config` / `creating_env` / `resetting` / `writing_obs` / `waiting_plan` / `stepping` / `stopped` / `completed` / `error`
- 在 runner 的关键阶段都刷新 `last_event_at`。
- 异常时写入 `error` 和 `traceback`。

**解决的问题：**

- 让 HUGSIM 子进程从“黑盒”变成可观测状态机，便于区分：
  - 卡在 reset
  - 卡在 env.step
  - 已把 obs 发出，正在等待 actor plan
  - 正常完成或异常退出

---

### 4. `framework/runner/env_factory.py`

**修复内容：**

- 将 `session_tag`、`fifo_step_timeout_s` 等 HUGSIM 运行时参数透传到环境构建逻辑。
- 让每个 actor 根据 `actor_id / worker_id / rank / cuda` 生成独立 session 标识。

**解决的问题：**

- 修复 HUGSIM session 隔离不完整，导致多个 actor 落到同一输出目录的问题。

---

### 5. `framework/rollout/collector.py`

**修复内容：**

- 增加单环境采样下的 `end_shard_on_done` / 提前结束 shard 行为。
- 对 HUGSIM 单 env actor，可在 episode 已结束时尽早收尾当前 shard，而不是无意义地继续 reset/补满。

**解决的问题：**

- 降低固定 horizon 下反复 reset 带来的额外耗时。
- 降低某些短 episode 场景的 reset 频率和累积阻塞。

---

### 6. `framework/runner/actor_runtime.py`

**修复内容：**

- 将 HUGSIM actor 的 `finish_shard_on_done` 行为接入 actor 主循环。
- 保持 learner 训练期间 actor pause / backpressure 机制不变，同时增强日志可解释性。

**解决的问题：**

- 让 actor 在 episode 已明确结束时更快释放当前 shard，减轻后续 reset 压力。

---

### 7. `framework/lightning/actor_learner_datamodule.py`

**修复内容：**

- 在之前排查过程中曾尝试“超时后允许部分 shard 更新”的 fault-tolerance 方案。
- 后续按需求恢复为：**learner 必须等待配置数量的 shard 才训练**。

**最终行为：**

- 本轮最终保留的是**严格等待 `shards_per_update`** 的行为，不再使用 partial shard workaround。

**说明：**

- 这部分不再是当前稳定性主修复手段，主要是回退到你要求的严格收集策略。

---

### 8. `framework/lightning/trajectory_datamodule.py`

**修复内容：**

- 将 `TrajectoryUpdateDataModule.train_dataloader()` 的 `num_workers` 从 `4` 改为 `0`。

**解决的问题：**

- learner 在 `build_training_batch()` 阶段会先把 `adv / ret / old_logp / obs_batch` 等张量放到 learner GPU。
- 当 DataLoader 再启用多 worker 进程时，worker 进程在 `__getitem__()` 中访问这些 GPU tensor，会触发：
  - `RuntimeError: CUDA error: initialization error`
- 改为 `num_workers=0` 后，由主进程直接迭代 GPU batch，避免 worker 进程重新初始化 CUDA。

## 本轮最关键的根因结论

### 根因 1：FIFO 边界协议不稳

旧实现依赖 writer 关闭 FIFO 才能让 reader 知道消息结束，这在长期持有 FIFO fd 的 actor/HUGSIM runner 场景下很脆弱，容易造成双向等待。

### 根因 2：session 生命周期未闭合

即使上层 wrapper 已判定 episode 结束，底层 FIFO runner 仍可能继续处于 `waiting_plan`，下一次 reset 时便形成“旧 episode 没真正结束，新 episode 又开始”的错位。

### 根因 3：learner DataLoader worker 访问 GPU batch

训练 batch 已提前搬到 learner GPU，但 DataLoader 仍开启多 worker，worker 进程访问 GPU tensor 时导致 CUDA 初始化错误，直接打断 learner。

## 当前结果

本轮修复后，主要现象已改善为：

- actor 不再因 FIFO 目录冲突互相踩会话文件。
- reset/step 的 FIFO 协议稳定性明显提升。
- wrapper 判终止后，底层 HUGSIM session 能同步关闭，不再长期遗留在旧状态。
- `status.json` 可用于区分正常 `waiting_plan` 与真实 reset/step 卡死。
- learner 的 CUDA DataLoader 初始化错误已规避。
- 当前训练观察为：**训练正常、reset 正常**。

---

## 后续建议

如果后续还要继续收敛稳定性，可优先关注：

1. learner 更新期间的 backpressure 是否偏强，导致 actor 长时间停在上层等待。
2. 某些特定 scenario 的 `resetting` / `stepping` 是否仍显著慢于其他场景。
3. 是否需要把 FIFO `status.json` 再增加 actor 侧联动字段，便于把 runner 状态和 actor 主循环状态一并看清。
